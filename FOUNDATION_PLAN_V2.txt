Locked Design Decisions
- MySQL legacy DB is the source of truth for user identity
- PostgreSQL is the source of truth for borrowing domain data
- No cross-database foreign keys
- default role = borrower
- local roles = borrower, staff, admin
- staff can manage assets
- 1 borrow request can contain multiple asset items
- first approved, first served
- pending requests do not deduct stock
- approve deducts stock
- V1 approve model = all-or-nothing per request
- partial return is supported
- return stock logic:
  - good -> available_qty += return_qty
  - damaged -> available_qty += return_qty
  - lost -> total_qty -= return_qty and available_qty does not increase
- overdue is a derived state, not a stored enum status
- serial_number is not part of assets V1
- returned_qty is computed from return_transaction_items, not stored on borrow_request_items
- request_no format = BR-{YYYY}-{0001..n}
- POST /api/users uses upsert semantic
- session model:
  - server-side session record stored in PostgreSQL
  - cookie stores signed session ID only
  - TTL = 8 hours
- OOP layering:
  - Route Handler -> Service -> Repository / Policy
- Route Handlers must not access DB directly
- Important state changes must write audit logs

Borrow creation
- create request header + items
- normalize duplicate assetId before persistence or reject by validation
- advisory stock check only
- no stock deduction on create

Approve flow
- all-or-nothing
- transaction:
  - load request and items
  - lock asset rows FOR UPDATE
  - verify every asset has enough availableQty
  - if any fail -> throw InsufficientStockError with insufficientItems details
  - deduct availableQty
  - set request status approved
  - set approvedQty = requestedQty for every item
  - write audit log

Reject flow
- pending only
- set rejected metadata and reason
- audit

Cancel flow
- pending only
- owner OR staff OR admin
- cancelReason required
- set cancelled metadata
- audit

Request detail/list rules
- returnedQty in API response is computed from return_transaction_items
- overdue flag is computed at query time when dueDate < current_date and status is approved or partially_returned
- support overdue=true filter in list endpoint

Rewrite API routes:
- POST /api/returns
- GET /api/returns
- GET /api/returns/[id]

Return flow
- only for approved or partially_returned requests
- transaction:
  - validate items belong to the request
  - validate cumulative returned qty <= approvedQty
  - insert return header + items
  - stock update by condition:
    - good -> available_qty += returnQty
    - damaged -> available_qty += returnQty
    - lost -> total_qty -= returnQty
  - if lost causes available_qty > total_qty, adjust available_qty down
  - recalculate request status
  - audit

Error rule
- if cumulative return exceeds approved -> throw ReturnExceedsApprovedError

Borrowing

Returns

- damaged return restores availableQty

Computed fields
- returnedQty in request detail is computed correctly
- overdue flag is computed correctly

Access

Audit and dashboard
